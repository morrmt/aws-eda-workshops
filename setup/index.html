    Getting Started¶

    Sign in to the provided AWS accounts¶

    At the beginning of the workshop you have been provided with a 12-character access code. This access code grants you permission to use a dedicated AWS account for the purposes of this workshop.

    Go to https://dashboard.eventengine.run/, enter the access code and click Proceed.

    On the User Dashboard, please click AWS Console to log into the AWS Management Console.

    Click Open Console. For the purposes of this workshop, you will not need to use command line and API access credentials.

    Overview of modules¶

    Time permitting, you will run the following hands-on labs. Start with the prerequisites.

    # Lab Module Recommendation Overview
    1 Prerequisites Required, start here Set up the lab environment and provision the prerequisite resources
    2 Deploy environment Recommended, for provisioned clusters Using CloudFormation, deploy an elastic computing environment on the AWS Cloud.
    3 Log into remote desktop Recommended, for provisioned clusters Clone an Aurora DB cluster and observing the divergence of the data set.
    4 Launch Vivado GUI Recommended, for provisioned clusters Backtrack an Aurora DB cluster to fix an accidental DDL operation.
    5 Submit Batch Workload Recommended, for provisioned clusters Examine the performance of your DB instances using RDS Performance Insights

    Lab environment at a glance¶

    To ensure everyone has a consistent experience, we have created a foundational template for AWS CloudFormation that provisions the resources needed for the lab environment.

    The environment deployed using CloudFormation includes several components:

    • a Amazon VPC network configuration with public and private subnets
    • a Database subnet group and relevant security groups for the cluster and workstation
    • a Amazon EC2 instance configured with the software components needed for the lab
    • IAM roles with access permissions for the workstation and cluster permissions for enhanced monitoring, S3 access and logging
    • Custom cluster and DB instance parameter groups for the Amazon Aurora cluster, enabling logging and performance schema
    • an Amazon Aurora MySQL DB cluster with 2 nodes: a writer and read replica
    • the master database credentials will be generated automatically and stored in an AWS Secrets Manager secret.
    • a read replica auto scaling configuration
    • a AWS Systems Manager command document to execute a load test
